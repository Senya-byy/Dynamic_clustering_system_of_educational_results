# PLAN_45_DAYS_TEAM (45 дней / 3 роли)

Роли:
- **Timlid** (`ShOtnik`) — интеграция, релизы, приоритеты, ревью
- **Architect** (`Egor`) — схема БД, API контракты, производительность, архитектурные решения
- **Tester** (`Anton`) — чек-листы, регресс, мобильный LAN, дефекты/приёмка

Цель 45 дней: стабилизировать сценарий **«QR → вопрос → ответ → проверка → рейтинг/аналитика/кластеризация»** и подготовить проект к демонстрации/учебному использованию.

## 1) Общие правила процесса
- Итерации: **6 недель + 3 дня**.
- Ритм: 3–5 коммитов/неделя, еженедельное демо и фиксируемый список дефектов.
- DoD для фичи:
  - API документировано (пример request/response),
  - UI проверено desktop + mobile,
  - учтены роли/права (teacher/student/admin),
  - smoke по ключевому сценарию проходит.

## 2) Критерии приёмки по неделям (Definition of Success)

### Неделя 1 — LAN/QR стабилизация
- Телефон открывает ссылку из QR **≤ 2 секунды** в одной Wi‑Fi сети.
- `nonce` живёт **~3.5 сек**, повторный/просроченный nonce даёт понятную ошибку.
- `device binding` работает: одно устройство не может «перепривязаться» к другому логину в рамках сессии.

### Неделя 2 — Проверка ответов и вопрос в проверке
- В проверке преподаватель видит: **вопрос, тему, сложность, max_score, ответ**.
- Оценивание сохраняется и отображается в «Мои ответы».

### Неделя 3 — Аналитика по вкладкам
- Разделение на вкладки (студенты/кластеризация/состав/переходы) без перегрузки страницы.
- Таблица студентов корректна при 0 ответов и частичных оценках.

### Неделя 4 — Кластеризация и история
- Запуск кластеризации создаёт `ClusterRun` и `ClusterAssignment`.
- Доступны `/runs`, `/runs/<id>`, `/transitions` и heatmap.
- Silhouette score отображается и сохраняется.

### Неделя 5 — Права/админка/удаления
- teacher видит только свои группы, student не видит чужие данные, admin видит всё.
- Удаления не оставляют «висячих» записей (answers/attendance/cluster runs и пр.).

### Неделя 6 — Документация/упаковка/предрелиз
- Есть runbook по LAN/QR, API examples, релизный чек-лист.
- Проект запускается локально и через docker-compose по README.

### Дни 43–45 — Буфер
- Закрыты все P0/P1 дефекты.
- Smoke + mobile regression пройдены повторно.

## 4) Риски и mitigation (расширено)
- **Неправильный IP в QR → телефон не открывает**  
  Mitigation: в UI добавить кнопку/подсказку «Показать сетевой URL», использовать `meta/qr-origin`, в runbook отдельный раздел диагностики.
- **Потеря данных при ручной миграции SQLite**  
  Mitigation: перед запуском миграций делать backup: копия файла БД (`cp classqr.db classqr.db.bak`), документировать в runbook.
- **Порты заняты (AirPlay/старые процессы)**  
  Mitigation: dev-скрипты проверяют порт и предлагают убить процесс/использовать альтернативный порт.
- **Телефон в другой Wi‑Fi сети**  
  Mitigation: явная ошибка + пункт в мобильном чек-листе «разные сети».
