## Личный дневник разработки ClassQR (ClassQR / Dynamic clustering system of educational results)

Документ фиксирует, что было сделано в ходе подготовки проекта к MVP/релизу, и какие проблемы встречались при деплое (PythonAnywhere + Render), с решениями.

### 2026-04-27 — Старт: постановка требований и ограничений

#### Исходная задача (функциональные требования)
- Система **ClassQR**: опросы/занятия через **QR**, выдача вопросов, приём ответов, проверка/оценка, рейтинг и аналитика.
- Ключевые доменные модули:
  - **Сессии**: создание пары/сессии, генерация QR, вход студентов по QR.
  - **Вопросы**: выдача одного вопроса или пула вопросов, рекомендации.
  - **Ответы**: приём ответов, хранение, проверка, выставление баллов/комментариев.
  - **Рейтинг**: вычисление рейтинга по группе.
  - **Кластеризация**: k-means на NumPy, расчёт силуэта, динамика кластеров.

#### Что требовалось сделать по качеству/рефакторингу
- Пересмотреть бизнес‑логику всех модулей, убрать нелогичные/избыточные шаги и дубли.
- Упростить цепочки вызовов, привести имена к единому стилю (PEP 8 для Python; рекомендации Vue для фронта).
- Добавить валидацию входных данных на backend (типы, диапазоны, обязательные поля).
- Оптимизировать k-means: ускорение вычислений + защита от вырожденных случаев (мало студентов, нулевая дисперсия и т.п.).

#### Подготовка к MVP/релизу
- Удалить отладочный код (print/console.log/закомментированные блоки).
- Добавить централизованную обработку ошибок (понятные HTTP‑статусы, логирование).
- Настроить CORS под production (разрешить только домен фронта).
- Подготовить конфиги сред (development/production), `.env.example`.
- Docker/Docker Compose: добавить/оптимизировать; убедиться, что всё стартует предсказуемо.
- Проверить идемпотентность миграций БД (для текущей SQLite-схемы — безопасные `ALTER TABLE`).
- Добавить `GET /health` для мониторинга.

#### Ограничения
- **Не менять C4-архитектуру**: контроллеры → сервисы → репозитории → модели.
- **Не ломать существующий API** для уже работающего фронта (обратная совместимость маршрутов/форматов).

#### Формат результата
- Пояснять после каждого изменения: зачем сделано и чем улучшает систему.
- Итог: список изменённых файлов, краткое описание, чек‑лист готовности к деплою.
- При архитектурных правках — диаграмма последовательности для сценария «студент отвечает на вопрос».

### 2026-04-27 — Рефакторинг и подготовка к MVP

#### 1) Инвентаризация архитектуры и API
- **Backend**: Flask + Flask-SQLAlchemy, слои C4 сохранены: контроллеры → сервисы → репозитории → модели.
- **Точки входа API**: маршруты регистрируются в `backend/app.py` через `add_url_rule`, auth — blueprint `/api/auth`.
- **Ключевые сценарии**: создание сессий/QR, выдача вопросов, приём ответов, проверка/оценка, рейтинг, аналитика и кластеризация.

#### 2) Централизованная обработка ошибок + логирование
- Добавлен модуль `backend/utils/error_handlers.py` и подключение в `backend/app.py`.
- Введены обработчики `HTTPException`, `ValueError`, `PermissionError`, общий `Exception`.
- **Эффект**: единый JSON-формат ошибок `{"error": "..."}`, меньше дублирования `try/except`, скрытие внутренних 500 от клиента, логирование stack trace.

#### 3) Healthcheck для мониторинга
- Добавлен эндпоинт `GET /health` в `backend/app.py` с лёгкой проверкой БД (`SELECT 1`).
- **Эффект**: можно использовать в Docker/оркестрации и при деплое для быстрой проверки доступности.

#### 4) CORS и конфигурация окружений (dev/prod)
- В `backend/app.py` добавлена логика `APP_ENV`/`FLASK_ENV`.
- **Safe-by-default**:
  - `production`: CORS не разрешает ничего, пока явно не задан `CORS_ORIGINS`.
  - `development`: разрешены localhost + опционально LAN.
- `CORS_ALLOW_LAN` по умолчанию выключен в production.
- Seed-данные (`seed_data`) по умолчанию отключены в production через `SEED_DATA`.
- `.env.example` обновлён: секции common/dev/prod + пример домена `https://SenyaByy.pythonanywhere.com`.

#### 5) Валидация входных данных в контроллерах
- Добавлен `backend/utils/validation.py` (require_json/get_int/get_str/get_bool).
- Применено без изменения публичных маршрутов и ключей JSON:
  - `backend/controllers/session_controller.py`
  - `backend/controllers/answer_controller.py`
  - `backend/controllers/question_controller.py`
- **Эффект**: меньше 500 из-за невалидных payload, больше понятных 400, единый стиль.

#### 6) Оптимизация кластеризации (NumPy k-means) и защита от вырождения
- Файл: `backend/services/clustering_service.py`.
- Оптимизации:
  - расчёт расстояний без временного массива `(n,k,d)` через формулу \(||x-c||^2\).
  - пересчёт центроидов через агрегации `np.bincount`.
  - early stop по стабилизации меток.
  - guards: `k>=n`, NaN/inf, “все точки одинаковые” (быстрый выход).
  - silhouette: дистанции считаются 1 раз; для больших групп — детерминированный subsample.
  - ускорено извлечение признаков: убрано N+1 по Attendance, батч‑загрузка Questions.
- **Эффект**: быстрее и устойчивее на классах/группах, меньше нагрузки на БД.

#### 7) Удаление отладочного кода
- Убраны `print` в `backend/utils/jwt_utils.py`, заменены на `logging`.
- Дополнительно: в проекте обнаружен `print` только в `backend/seed_math_demo.py` (это демо-скрипт, не используется в рантайме API).

#### 8) Docker/Compose и релизная упаковка
- `backend/Dockerfile` приведён к порту **5001** (в соответствии с compose).
- В `docker-compose.yml` и `scripts/docker-compose.yml` добавлены:
  - `APP_ENV`, `SEED_DATA`, prod-ориентиры для `CORS_ORIGINS/QR_FRONTEND_ORIGIN`.
  - healthcheck для backend (`/health`) и nginx.
- Идемпотентные SQLite-миграции расширены в `_ensure_sqlite_migrations()` (добавлены guards для дополнительных колонок).

#### 9) Тесты/проверки
- Исправлен импорт и изоляция окружения в `backend/tests/test_auth.py`.
- Выявленная проблема: при автосидинге и особенностях инициализации приложения тесты ловили `UNIQUE constraint failed: users.login`.
- Решение:
  - тестовый режим `APP_ENV=test` отключает auto-create/seed внутри `create_app`.
  - модульный `app = create_app()` не создаётся при `APP_ENV=test` (чтобы не было побочных эффектов на import).
  - в фикстуре добавлена защитная очистка `testuser`.
- Итог: `pytest` проходит успешно (3 passed).

### 2026-04-27 — Деплой на PythonAnywhere (backend + подготовка frontend)

#### Проблема 1: нет `python` в PATH локально для тестов
- В локальной среде вызов `python` не работал, использовали `python3`.

#### Проблема 2: ModuleNotFoundError: numpy
- Причина: на PythonAnywhere зависимости ставились из корневого `requirements.txt`, в котором не было `numpy` (он был в `backend/requirements.txt`).
- Решение: `pip install -r backend/requirements.txt`.

#### Проблема 3: почему `/` отдаёт 404
- Причина: backend — это API, роут `/` не определён.
- Проверка правильности запуска: `GET /health` → `status=ok`.

#### Проблема 4: нет npm на PythonAnywhere для сборки Vue
- `npm: command not found`.
- Попытка поставить Node через `nodeenv` привела к сломанному npm (`Cannot find module 'are-we-there-yet'`).
- Рекомендованный обход: собирать `frontend/dist` локально и загружать как static.

#### Настройка WSGI
- Добавлен `backend/wsgi.py` (переменная `application` для PythonAnywhere/WSGI).
- Добавлена инструкция `docs/DEPLOY_PYTHONANYWHERE.md`.

### 2026-04-27 — Деплой на Render (полностью браузерный вариант)

#### Проблема 1: Render использовал Python 3.14 по умолчанию, падала сборка Pillow
- Симптом: `Using Python version 3.14.3 (default)` и падение сборки `Pillow==10.2.0` (не было wheel).
- Попытка: добавить `runtime.txt` (в root и `backend/`) с `python-3.11.9`.
- Итог: Render всё равно тянул 3.14 → переход на Docker.

#### Решение: Backend как Docker Web Service
- Использован `backend/Dockerfile` на `python:3.11-slim` (уходит проблема 3.14).
- Важно было не “удвоить путь”:
  - ошибка `backend/backend` возникала при одновременном `Root Directory=backend` и `Dockerfile Path=backend/Dockerfile`.
  - исправление: либо Root Directory пусто + Dockerfile Path `backend/Dockerfile`, либо Root Directory `backend` + Dockerfile Path `Dockerfile`.
- Проверка: `GET /health` на Render → `ok`.

#### Frontend на Render Static Site
- Важная ошибка: `Empty build command; skipping build` → `dist does not exist`.
- Решение: задать build команду `npm ci && npm run build`, publish directory `dist`, root directory `frontend`.

#### Связь frontend ↔ backend (API baseURL)
- По коду фронт ходил на `/api` (dev proxy решал локально).
- Для Render добавлена поддержка `VITE_API_BASE`:
  - `frontend/src/api/index.js`: baseURL берётся из `import.meta.env.VITE_API_BASE` или fallback `/api`.
- Для backend добавлен `gunicorn` в `backend/requirements.txt` (для корректного запуска на хостинге).

#### Проблема: 401 Invalid credentials
- Это означало, что связь фронт↔бэк есть, но нет пользователя в БД.
- В production seed выключен (`SEED_DATA=false`), поэтому дефолтные пользователи не создаются.

#### Bootstrap администратора без seed (инициализация первого admin)
- Потребность: создать первого admin без shell.
- Реализовано:
  - `POST /api/admin/bootstrap` (работает только если задан env `BOOTSTRAP_TOKEN` и при наличии заголовка `X-Bootstrap-Token`).
  - Создаёт admin только если админа ещё нет.
- Рекомендация: после создания администратора удалить `BOOTSTRAP_TOKEN` из env и перезапустить сервис (endpoint станет 404).

### Нерешённые/важные замечания (технический долг)
- На Render **SQLite** может быть непредсказуем (файловая система не гарантирует сохранность на free). Для реального прод лучше Postgres.
- В проекте есть **дублирование JWT утилит** (`middleware/auth_middleware.py` и `backend/utils/jwt_utils.py`) с разными payload-ключами (`id` vs `user_id`). Сейчас это не ломает поток, но опасно при дальнейшем расширении.
