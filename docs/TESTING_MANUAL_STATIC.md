# TESTING_MANUAL_STATIC — статическое и ручное динамическое тестирование ClassQR

Документ описывает, как проверить проект **ClassQR** без автотестов:  
- **статическое тестирование** (без запуска, по коду/конфигам)  
- **ручное динамическое тестирование** (запуск и прогон сценариев)

Стек проекта:
- Frontend: Vue 3 + Vite + Pinia + Axios + Vue Router (history mode)
- Backend: Python + Flask (REST API)
- DB: PostgreSQL (на Render) / SQLite (локально)

---

## 1) Подготовка окружения для ручного тестирования

### 1.0 Параметры вашего Render-окружения (заполнить один раз)

Ниже — блок, который удобно скопировать в отчёт по тестированию.

- **Frontend (Render Static Site)**: `https://classqr-frontend.onrender.com`
- **Backend (Render Web Service)**: `https://<ваш-backend>.onrender.com`
- **DB**: Render PostgreSQL `classqr-db`
- **Seed**: обычно `SEED_DATA=false` (значит “дефолтных” пользователей может не быть)

### 1.1 Данные/аккаунты
Нужны минимум 3 роли:
- **admin**
- **teacher**
- **student** (желательно 3+ студентов для кластеризации)

Если в прод окружении seed выключен (`SEED_DATA=false`), заранее подготовьте тестовые сущности:
- группа, привязанная к преподавателю
- студенты в группе
- темы/вопросы (разной сложности и с `max_score`)

Если пользователей ещё нет (частый случай на новой Render Postgres БД):
- создайте первого admin через bootstrap endpoint `POST /api/admin/bootstrap` (см. `ЛичныйДневник.md`, раздел про Render)
- после создания admin: зайдите в админку и заведите teacher/student и тестовые данные

### 1.2 URL-ы окружений
Зафиксируйте:
- **Frontend URL** (Render Static Site): `https://classqr-frontend.onrender.com`
- **Backend URL** (Render Web Service): `https://<ваш-backend>.onrender.com`

Для локального окружения:
- frontend: `http://127.0.0.1:5173`
- backend: `http://127.0.0.1:5001`

### 1.3 Инструменты
- Браузер + DevTools (Console/Network)
- `curl` (или Postman/Insomnia)
- (опционально) 2 устройства: ноутбук (teacher) + телефон (student) для QR/LAN сценариев

## 2) Статическое тестирование (до запуска)

Цель: быстро найти “ошибки конфигурации” и логические несостыковки до прогона UI.

### 2.1 Проверка документации и окружения
- Убедиться, что в `README.md` есть актуальные шаги запуска/деплоя.
- Для Render Static Site: в настройках включён SPA rewrite  
  `/*` → `/index.html` (**Rewrite**), иначе при F5 будет 404 на маршрутах vue-router.
- Проверить `.env.example`:
  - есть `JWT_SECRET_KEY`, `SECRET_KEY`
  - есть `DATABASE_URL`
  - есть `CORS_ORIGINS`, `QR_FRONTEND_ORIGIN`

### 2.2 Проверка backend (логика/безопасность)
По коду (быстрое ревью):
- **Auth/JWT**: защищены ли все эндпоинты, которые не должны быть публичными.
- **RBAC**: проверяются ли роли для teacher/admin/student (особенно: ответы, сессии, админка).
- **Валидация входа**: обязательные поля, типы, пустые строки.
- **Ошибки**: формат `{ "error": "..." }` и корректные HTTP статусы.
- **CORS**: нет ли `*` в production, корректно ли настроены origins под фронт.
- **DB**: нет ли кода, который может случайно “дропнуть/пересоздать” таблицы в проде.

### 2.3 Проверка frontend (роутинг/конфиги)
- Vue Router: используется `createWebHistory()` → значит на хостинге должен быть rewrite на `index.html`.
- Проверить, как фронт выбирает базовый URL API:
  - локально через `/api` (прокси Vite)
  - в проде через переменную окружения (например `VITE_API_BASE`) — если это реализовано в проекте
- Проверить, что в UI есть обработка ошибок (показываются понятные сообщения, а не “тихий” провал).

## 3) Ручное динамическое тестирование (основной прогон)

### 3.1 Smoke: сервисы живы
- **Backend healthcheck**:
  - `GET /health` → `200`
- **Frontend**:
  - открыть главную страницу
  - обновить страницу (F5) на **любой внутренней странице** (например `/teacher/sessions`) — не должно быть 404

### 3.2 Auth: логин/выход/сессия
Проверить для каждой роли:
- логин проходит, в ответе/хранилище появляется `access_token` и роль
- обновление страницы (F5) не ломает состояние (редиректы корректны)
- попытка зайти без токена на защищённые страницы → редирект на `/login`

Негатив:
- неверный пароль → понятная ошибка
- истёкший/битый JWT → 401 и редирект/релогин

### 3.3 Teacher: сессии + Live QR
Сценарий:
- зайти teacher
- открыть список сессий
- создать сессию (один вопрос)
- включить “Живой QR”
- убедиться, что QR/nonce **обновляются** (nonce меняется, `expires_in_seconds` небольшой)

Негатив:
- открыть QR “старый” nonce (подождать 5–10 секунд) → ошибка “просрочен”

### 3.4 Student: вход по QR (verify-ticket) + закрепление вопроса
Сценарий:
- зайти student
- открыть ссылку `/join?code=...&nonce=...` (из QR)
- verify-ticket успешен, пришёл вопрос
- повторить verify-ticket для этой же сессии (или обновить `/join`) → студент **не должен** получить новый вопрос повторно (ожидаем бизнес-ошибку)

Негатив:
- verify-ticket без `device_id` или с пустым `device_id` → ошибка 400

### 3.5 Student: отправка ответа (submit)
Сценарий:
- отправить ответ → `201 Created`
- убедиться, что ответ появился в “Мои ответы”

Негатив:
- повторная отправка ответа в ту же сессию → отказ (400)
- отправка после таймера (если включён таймер) → `is_late=true`

### 3.6 Teacher: проверка ответов (grade)
Сценарий:
- teacher открывает “Проверка”
- видит ответы студентов
- в карточке ответа отображается вопрос (текст/тема/сложность/`max_score`)
- выставить оценку/комментарий/правильность → сохранить
- обновить страницу → оценка сохранена

Негатив:
- teacher пытается открыть/оценить ответы “не своей” группы → 403

### 3.7 Rating
Сценарий:
- student видит рейтинг своей группы
- teacher/admin тоже видят рейтинг
- если включён privacy_mode — в рейтинге скрывается имя/детали корректно (проверить `is_hidden`)

### 3.8 Analytics / Clustering
Предусловие: 3+ студентов с достаточным количеством оценённых ответов.

Сценарий:
- teacher/admin открывает аналитику
- вкладка “студенты” показывает метрики
- запустить кластеризацию → создаётся run (есть `n_clusters`, `silhouette_score`)
- открыть историю запусков (`/runs`)
- transitions/heatmap отображаются без ошибок

Негатив:
- попытка кластеризации при <3 студентах → понятная ошибка 400

## 4) Динамическое тестирование API (через curl/Postman)

Цель: независимо от UI проверить контракты и статусы.

Рекомендуемые проверки:
- `POST /api/auth/login` (teacher/student/admin)
- teacher:
  - `GET /api/sessions/teacher`
  - `GET /api/sessions/{id}/live-qr`
  - `GET /api/sessions/{id}/answers`
  - `POST /api/answers/{id}/grade`
- student:
  - `POST /api/sessions/verify-ticket`
  - `POST /api/answers/submit`
- analytics:
  - `POST /api/analytics/cluster/{group_id}`
  - `GET /api/analytics/cluster/{group_id}/transitions`

Сверяйтесь с `docs/API_EXAMPLES.md`.

---

## 5) Нагрузочное/устойчивость (минимум вручную)

Без сложных тулов можно проверить “краевые” вещи:
- несколько студентов одновременно отправляют ответы в одну сессию
- teacher несколько раз подряд включает live-QR (не накапливаются ли “лишние” join tickets)
- повторные запросы (двойной клик “Отправить”) не приводят к двойным записям

---

## 6) Итоговый отчёт (что сдавать как результат тестирования)

Сформируйте короткий отчёт:
- окружение (Render URLs / локально), дата/версия
- что проверяли (разделы 3.1–3.8)
- список найденных дефектов:
  - шаги воспроизведения
  - ожидаемое/фактическое
  - скриншот/лог из DevTools Network/Console
  - критичность (blocker/major/minor)
