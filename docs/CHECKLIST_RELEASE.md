# CHECKLIST_RELEASE — регрессионный чек-лист (20 пунктов)

Цель: перед релизом убедиться, что ключевые сценарии работают для всех ролей и на телефоне в LAN.

## A. Запуск и окружение
1. Backend стартует без ошибок, доступен `GET /api/auth/profile` (после логина).
2. Frontend стартует, доступен `http://<IP>:5173` (есть Network URL).
3. В QR не попадает `localhost` (проверить `join_url` в ответе `live-qr`).
4. CORS_ALLOW_LAN включён (телефон может выполнять API запросы через фронт).

## B. Teacher: сессии и QR
5. Teacher логинится и видит список своих групп (`GET /api/groups`).
6. Teacher создаёт сессию (один вопрос) и видит её в списке.
7. Teacher создаёт сессию (пул по темам), сессия создаётся успешно.
8. Teacher включает «Живой QR», QR обновляется раз в секунду, `nonce` меняется.

## C. Student: вход по QR, вопрос, ответ
9. Student сканирует QR, открывается `/join` и проходит verify-ticket.
10. Student получает вопрос, не может получить второй вопрос (повторный verify-ticket должен дать ошибку).
11. Student отправляет ответ (`answers/submit`) — получает 201, ответ появляется в проверке.
12. Негатив: повторная отправка ответа в ту же сессию → ошибка «Ответ уже отправлен».
13. Негатив: просроченный nonce → ошибка «просроченный QR».
14. Негатив: device bound: попытка войти другим логином на том же устройстве → понятная ошибка.

## D. Teacher: проверка
15. Teacher открывает «Проверка», выбирает сессию, видит ответы студентов.
16. В карточке ответа отображается **вопрос**: текст, тема, сложность, max_score.
17. Teacher ставит оценку, сохраняет; обновление `checked_at` видно.
18. Негатив: teacher пытается открыть ответы чужой группы → 403.

## E. Rating
19. Teacher/Student видят рейтинг группы, приватность соблюдается (privacy_mode).

## F. Analytics / Clustering
20. В аналитике:
   - вкладка студентов показывает метрики,
   - кластеризация запускается при ≥3 студентах,
   - есть история `/runs`,
   - transitions отображает heatmap и доли кластеров.
