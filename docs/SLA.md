# SLA — соглашение об уровне сервиса (ClassQR)

Документ задаёт **ожидаемый уровень сервиса** для системы ClassQR (учёт посещаемости, ответы на парах, QR/PIN-вход, аналитика).  
Формат: **внутренний / учебный проект** — не юридический договор с заказчиком, а рамка для планирования, приёмки и общения с пользователями (преподаватели, студенты, администраторы).

**Стороны (условно)**:

| Роль | Ответственность |
|------|------------------|
| **Поставщик сервиса** | владелец/команда разработки, деплой, исправления, коммуникация о сбоях |
| **Потребители** | преподаватели, студенты, администраторы инстанса |

---

## 1. Объект SLA (что входит в «сервис»)

В зону действия входят:

- **Backend** API (Flask), доступный по HTTPS в production.
- **Frontend** SPA (Vue), раздаётся как статика.
- **База данных** PostgreSQL (инстанс, привязанный к приложению).
- Критические пользовательские сценарии: вход, создание пары/сессии, живой QR и PIN-вход, получение вопроса, отправка ответа, проверка ответов преподавателем, базовые функции админки.

Не входят (или входят с пониженными обязательствами):

- сторонние сети (Wi‑Fi аудитории, DNS, браузеры на устройствах студентов);
- рекомендации/внешние сервисы, если они подключены опционально;
- экспериментальные разделы (например, часть аналитики), если явно помечены как «бета» в интерфейсе или документации.

---

## 2. Целевые показатели доступности (SLO)

Показатели ориентировочные; фактическая доступность зависит от тарифа **Render** и конфигурации инстанса.

| Уровень | Целевой аптайм API* | Комментарий |
|---------|---------------------|---------------|
| **MVP / учебная эксплуатация** | не ниже **95%** в месяц | измеряется по `/health` или внешнему пингу; простои планового обновления учитываются отдельно (п. 6) |
| **Желательный** | **99%** | при выделенном инстансе без «усыпления», мониторинге и резервировании БД |

\*Окно измерения: календарный месяц. Исключения — п. 7.

**Frontend**: считается доступным, если статика отдаётся CDN/хостингом Render без массовых 5xx; локальная разработка (localhost) в SLA не входит.

---

## 3. Время отклика (латентность)

Цели **не жёсткие**, для мониторинга и регрессии:

| Операция | Целевое время (p95)** |
|----------|------------------------|
| Типичный API-запрос (CRUD, без тяжёлой аналитики) | ≤ **2 с** |
| Вход, verify-ticket / verify-pin, submit answer | ≤ **3 с** |
| Отчёты / кластеризация по большой группе | ≤ **30 с** или асинхронный сценарий с индикацией загрузки |

**p95** — 95-й перцентиль при нормальной нагрузке учебной группы (десятки одновременных пользователей), не нагрузочное тестирование.

---

## 4. Надёжность данных (RPO / RTO)

| Параметр | Цель |
|----------|------|
| **RPO** (максимально допустимая потеря данных) | не более **24 ч** без бэкапа БД; при включённых автоматических бэкапах Render — по политике провайдера |
| **RTO** (время восстановления после критического сбоя) | **≤ 8 ч** рабочего времени команды на MVP; **≤ 4 ч** при наличии runbook и актуального бэкапа |

Рекомендация: зафиксировать в `docs/RUNBOOK_*` порядок восстановления из бэкапа и переменных окружения.

---

## 5. Поддержка и каналы связи

| Канал | Назначение |
|-------|------------|
| согласованный мессенджер / почта курса | инциденты, вопросы преподавателей |
| Issues в репозитории (если используется) | баги, пожелания, приоритизация |

**Рабочее время ответа** (не время решения):

| Критичность | Описание | Первый ответ |
|-------------|----------|--------------|
| **P1** | сервис недоступен для большинства пользователей | **≤ 4 ч** в учебные дни |
| **P2** | деградация (часть сценариев, обходной путь есть) | **≤ 1 рабочий день** |
| **P3** | косметика, пожелания | по плану релизов |

«Рабочие дни» — по календарю учебной команды (указать в README или у преподавателя курса).

---

## 6. Плановые работы и изменения

- Плановые обновления **не чаще 1 раза в неделю** вне окон экзаменов/зачётов (или по согласованию).
- Уведомление за **≥ 24 ч**, если ожидается простой **> 15 минут**; краткие рестарты (< 15 мин) — по возможности заранее в общем чате.
- Критические патчи безопасности — **внепланово**, с постфактум-уведомлением.

---

## 7. Исключения из SLA (не учитываются в аптайме)

- сбои **Render**, провайдера DNS, магистральных сетей;
- действия пользователей (неверный PIN, просроченный nonce, блокировки браузера);
- форс-мажор;
- превышение лимитов бесплатного/дешёвого тарифа (sleep инстанса, исчерпание квот) — до момента апгрейда или перенастройки;
- работы по согласованию с заказчиком, выходящие за рамки согласованного MVP.

---

## 8. Метрики и отчётность

- **Минимум**: периодическая проверка `GET /health` (в т.ч. `db: ok`), ручной smoke по `docs/MIN_TESTS.md` или аналогу после релиза.
- **Желательно**: простой аптайм-чекер (cron / внешний сервис) с алертом в мессенджер при падении API **> 15 минут**.

Раз в **квартал** (или после учебного модуля) — короткий разбор: фактический аптайм, инциденты, изменения в SLA при смене тарифа или команды.
